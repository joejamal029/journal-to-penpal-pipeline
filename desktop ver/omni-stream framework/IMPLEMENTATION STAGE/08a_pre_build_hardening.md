[PROMPT 8a: PRE-BUILD HARDENING]
System Instruction: Activate Documentation Audit Protocol (Pre-Build Quality Gate)

CONTEXT:
Phase 7 is complete. The build scaffold (CLAUDE.md, plan.md, memory.md, progress.txt, LESSONS.md) and the full spec suite (PRD.md, APP_FLOW.md, TECH_STACK.md, etc.) are generated and approved. Before handing the documentation suite to the build agent, this one-time hardening pass catches documentation-level defects that would otherwise become code-level bugs.

This prompt is run by the DOCUMENTATION AGENT — the same agent that produced Phases 1-7. The documentation agent wrote these docs; it should audit them.

WHY THIS EXISTS:
In observed builds, 39% of all post-production bugs were schema/column mismatches that originated as documentation gaps — missing columns, typos in column names, NOT NULL constraints without corresponding INSERT coverage, and enum definitions missing required values. These are not implementation failures. They are documentation defects that propagated into code. This phase catches them at the source.

But 39% is what has been measured. The true rate of documentation-originated bugs is likely higher, and the categories of documentation defects extend beyond schema mismatches. The audit steps below are the **known baseline** — they target the failure categories that have been observed. The auditing agent MUST also apply judgment: "What else could be wrong with these documents that no checklist item covers?" If the answer isn't "nothing," investigate and flag it.

## STEP 1: SCHEMA-CODE CONTRACT AUDIT

For every table defined in BACKEND_STRUCTURE.md:

1. **Column Completeness:**
   - Verify every column has an explicit type (DB-native: TEXT, INTEGER, REAL, BLOB — not "string" or "number")
   - Verify every NOT NULL column either has a DEFAULT value or is listed as required in INSERT operations
   - Verify every column referenced in an index actually exists in the table definition

2. **Foreign Key Integrity:**
   - For every REFERENCES clause: verify the target table and target column exist
   - Verify referential actions (CASCADE, SET NULL, RESTRICT) are specified where tables have parent-child relationships

3. **Enum Consistency:**
   - For every CHECK constraint that restricts values to a set (e.g., `CHECK(state IN ('pending','complete','missed'))`)
   - Verify that a corresponding TypeScript/code-level enum or union type is defined that has ALL the same values
   - Flag any code-level enum that adds or omits values compared to the DB CHECK constraint

4. **State Machine Completeness:**
   - For every entity with a `state` or `status` column:
   - Verify that ALL transitions are documented (from → to)
   - Verify there are no orphan states (states with no outgoing transitions AND no incoming transitions)
   - Verify terminal states are explicitly identified

5. **Timestamp Audit:**
   - For every table with `created_at` or `updated_at` columns:
   - Verify whether they are NOT NULL
   - Flag any table that has `updated_at NOT NULL` — this is a common source of INSERT failures (the insert must include it)

6. **Proactive Schema Analysis:**
   - After running checks 1-5, step back and think: "What else could be wrong with this schema that these checks didn't cover?"
   - Consider: tables with no obvious consumer, columns with mismatched semantics (name suggests one thing, type says another), missing indexes on columns used in WHERE clauses, tables designed for growth that have no pagination strategy, composite unique constraints that don't match the business rules
   - Flag anything found with the same severity treatment as the baseline checks

## STEP 2: CROSS-DOCUMENT CONSISTENCY CHECK

Verify that all documents agree with each other:

1. **FEAT-ID Coverage:**
   - Every FEAT-ID in PRD.md appears in at least one plan.md task
   - Every plan.md task has a valid FEAT-ID that exists in PRD.md
   - No orphaned FEAT-IDs in either direction

2. **Schema ↔ App Flow:**
   - Every table in BACKEND_STRUCTURE.md is referenced by at least one screen in APP_FLOW.md's data requirements
   - Every data requirement in APP_FLOW.md traces to a table or API endpoint in BACKEND_STRUCTURE.md
   - Flag any screen that claims data from a source not defined in BACKEND_STRUCTURE.md

3. **Tech Stack Completeness:**
   - Every library mentioned anywhere in the suite has a pinned version in TECH_STACK.md
   - No library in TECH_STACK.md is unused (not referenced by any other doc)
   - Flag any library with a version range (e.g., "^3.x") instead of a pinned version (e.g., "3.2.1")

4. **UX State Coverage:**
   - Every screen in APP_FLOW.md has all four states documented: Loading, Success, Empty, Error
   - Flag any screen missing one or more states

5. **Plan Task Quality:**
   - Every task in plan.md has: FEAT ref, Atlas pattern, Verify criterion, Depends on
   - Every task's "Verify" criterion is specific enough to automate (not "it works" but "test X passes" or "screen renders Y")
   - Flag any verify criterion that is vague or subjective

6. **Hard Rules ↔ Source:**
   - Every hard rule in CLAUDE.md traces to a named source (Crucible decision, Atlas constraint, or spec doc)
   - Flag any hard rule without a traceable source — it may be hallucinated

7. **Proactive Consistency Analysis:**
   - After running checks 1-6, step back and think: "Are there any cross-document inconsistencies that these structured checks didn't catch?"
   - Consider: implicit assumptions one document makes that another document contradicts, naming inconsistencies (same concept called different things in different docs), ordering assumptions (one doc assumes X happens before Y, another doc's flow implies Y first), features whose acceptance criteria are impossible to verify with the defined API endpoints
   - Flag anything found with the same severity treatment as the baseline checks

## STEP 3: ANTI-PATTERN INJECTION

Based on the project's tech stack (from TECH_STACK.md), generate stack-specific anti-patterns and prepare them for injection into CLAUDE.md §Anti-Pattern Rules.

### Detection Logic:

**If stack includes SQLite:**
- "Verify column names against schema.sql before AND after writing any SQL string"
- "All INSERT statements must include every NOT NULL column without DEFAULT values"
- "SQLite enforces CASE-SENSITIVE foreign keys — verify case matches between parent and child tables"

**If stack includes React / React Native:**
- "Never render a component with an event handler that is an empty function, console.log, or TODO comment"
- "Never use inline styles for values that exist as tokens in DESIGN_SYSTEM.md"
- "Every useState must have a corresponding cleanup or the state must be explained in a comment"

**If stack includes TypeScript:**
- "Every enum must contain ALL values that appear in corresponding database CHECK constraints"
- "No `any` type. Use `unknown` at system boundaries with type guards"
- "Every function called across file boundaries must have its parameter order verified against the source definition"

**If stack includes Zustand / Redux / state management:**
- "Every store mutation must call `set()` — computing new state without persisting it is an invisible no-op"
- "Never duplicate state across stores — one source of truth per data entity"

**If stack includes API / REST / fetch:**
- "Verify API endpoint URIs contain no embedded spaces, line breaks, or concatenation errors"
- "Every fetch call must handle the error case — no fire-and-forget network requests"

**If stack includes Expo / React Native:**
- "Verify all imports from expo-* packages match SDK version in TECH_STACK.md — breaking API changes between SDK versions are common"
- "Web platform requires async database operations — never assume synchronous DB access"

**Beyond the known stacks:**
The stack-specific patterns above are the **known baseline** — derived from projects that have used these technologies. If the project uses a technology not listed above, DO NOT assume it has no anti-patterns. Research common pitfalls for the project's specific stack and generate anti-patterns at the same level of specificity. Additionally, consider cross-stack interaction anti-patterns — issues that arise not from individual technologies but from how they interact (e.g., React Native + SQLite + Expo together produce platform-specific behavior that none of them exhibit alone).

OUTPUT: `_build_prep/anti_patterns.md` — to be appended to CLAUDE.md §Anti-Pattern Rules
